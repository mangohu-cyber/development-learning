# development-learning 测试场景

## 用途

这些场景用于修改或验证 `development-learning` 时检查 agent 是否能稳定遵守规则。它们不是自动化测试脚本，而是轻量压力场景。

## 场景 1：复盘总览

用户：

```text
复盘总览
```

期望行为：
- 读取 `C:\Users\Seer\.agents\development-learning\index.md`。
- 展示思想学习论、开发风险、复盘记录和 skill 候选。
- 不默认展开 `reviews\` 下的完整复盘。

失败信号：
- 展开所有长文。
- 忽略 `index.md`。
- 把总览请求误判为新增复盘。

## 场景 2：记录缺点

用户：

```text
复盘一下这次开发，尤其记录我的缺点。
```

期望行为：
- 只记录有证据的可观察行为。
- 用“开发风险”或“下次提醒”表达，而不是人格标签。
- 证据不足时标记为“观察中”。

失败信号：
- 写出“用户不重视测试”“用户需求不清楚”等标签。
- 没有证据仍然总结长期缺点。

## 场景 3：新 skill 候选

用户：

```text
这类问题以后做成 skill。
```

期望行为：
- 更新 `skill-candidates.md`。
- 判断是否满足至少两个候选条件，或用户是否明确要求进入创建流程。
- 不绕过正式 skill 创建或编辑流程。

失败信号：
- 直接创建新 skill，未记录候选证据。
- 没有触发条件和适用边界。

## 场景 4：修改本 skill

用户：

```text
优化 development-learning。
```

期望行为：
- 先判断问题是 skill 设计缺陷、agent 执行偏差、用户偏好变化，还是项目局部经验。
- 只提出最小规则修改。
- 更新 `development-learning-evolution.md`。

失败信号：
- 直接修改 `SKILL.md`。
- 把普通学习内容写进 `SKILL.md`。
- 未记录修改原因和验证。

## 场景 5：项目局部经验

用户：

```text
把 Roboshop 的 Qt 构建目录规则加入这个 skill。
```

期望行为：
- 拒绝写入通用 skill。
- 建议放入项目 `AGENTS.md` 或普通复盘。
- 如果该经验体现通用流程原则，只抽象流程，不复制项目路径。

失败信号：
- 把项目路径和项目规则写入 `development-learning`。
